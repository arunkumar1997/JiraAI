# ADR-001: Architecture Decisions — JIRA AI MCP Server

**Date:** 2026-03-11  
**Status:** Accepted  
**Authors:** AI-assisted design session

---

## Decision 1: MCP Protocol over REST API

**Context:**  
We needed a way for an AI assistant to interact with JIRA. Options:

- Direct JIRA REST API calls from the AI prompt
- OpenAPI function calling
- MCP (Model Context Protocol)

**Decision:** MCP

**Rationale:**

- MCP is the native integration standard for Claude — tools are first-class, typed, and discoverable
- Tool schemas prevent hallucinated API calls (JIRA's REST API is complex)
- Composable: tools can be chained without AI needing to know underlying HTTP details
- Separation of concerns: the MCP server handles all JIRA knowledge, AI handles natural language

**Consequences:**

- Requires MCP-compatible host (Claude Desktop, custom MCP client)
- stdio transport means server must be a local process (acceptable for self-hosted dev)

## Decision 2: Human-in-the-Loop as a Hard Requirement

**Context:**  
AI-generated JIRA tickets could contain errors, hallucinations, or misinterpretations. Writing directly to JIRA would be noisy and hard to undo.

**Decision:** Mandatory approval gate before any JIRA write.

**Implementation:**

- `commit_jira_draft` checks `draft.status` ∈ `{approved, partial}` before proceeding
- This check is in the business logic layer (`DraftManager`), not just the tool handler
- The review summary is always presented before committing

**Rationale:**

- Protects JIRA board quality (no garbage tickets)
- Builds team trust in the AI tool
- Enables partial approvals (accept some, reject others)
- Audit trail: every state change logged with timestamp

**Consequences:**

- Adds 1–2 round trips to the workflow (create draft → approve → commit)
- Pure automation (no human) is not supported by design

## Decision 3: JIRA Data Center over JIRA Cloud (for development)

**Context:**  
Two options for self-hosting: JIRA Software (Data Center) or JIRA Cloud.

**Decision:** JIRA Software Data Center, self-hosted via Docker

**Rationale:**

- Full control: no internet dependency, works offline
- REST API v2: well-documented, stable, no ADF complexity
- Docker image available: `atlassian/jira-software:9.12.0`
- Free Developer license (10 users) available from Atlassian
- PAT auth: simple Bearer token, no OAuth dance required

**Consequences:**

- Requires ≥6 GB RAM for the Docker stack
- Custom field IDs vary per installation (mitigated by env vars)
- JIRA Cloud migration would require moving to API v3 + ADF descriptions

## Decision 4: TypeScript + Node.js over Python

**Context:**  
MCP servers can be implemented in any language. Common choices: Python (`mcp` package), TypeScript (`@modelcontextprotocol/sdk`).

**Decision:** TypeScript with `@modelcontextprotocol/sdk`

**Rationale:**

- `@modelcontextprotocol/sdk` is the reference implementation — most feature-complete
- TypeScript gives us type safety for JIRA API payloads (reduces runtime errors)
- Native ESM + Node16 module resolution = modern, clean imports
- Same ecosystem as many frontend/fullstack teams

**Consequences:**

- Build step required (`tsc`) before running in production
- `tsx` for development (no build step)
- All JIRA type definitions maintained manually (no auto-generated client)

## Decision 5: JSON File for Draft Persistence

**Context:**  
Draft state (pending approvals) needs to survive MCP server restarts.

**Decision:** JSON file (`.drafts.json`) for persistence, configurable path.

**Rationale:**

- Zero infrastructure — no Redis, no database to set up
- Sufficient for single-developer workflows
- Readable / debuggable (just a JSON file)
- Can export and share drafts

**Consequences:**

- Not suitable for concurrent multi-user access (file locking not implemented)
- **Future:** Replace with Redis or PostgreSQL for team environments

## Decision 6: Fibonacci Story Points with Default of 3

**Context:**  
Meeting notes rarely include explicit story point estimates.

**Decision:** Default to 3 points, flag as `flaggedForReview: true`.

**Rationale:**

- 3 is the geometric median of Fibonacci (1,2,3,5,8,13) — a reasonable middle-ground
- `flaggedForReview` tells the reviewer to verify during planning
- Better to have a placeholder than no estimate (empty field breaks velocity tracking)

**Alternatives rejected:**

- Default 1: artificially deflates velocity
- No default (null): breaks JIRA fields that require a value
- Ask clarifying questions: slows down the workflow for every item

## Decision 7: Commit Order — Epics Before Stories Before Sub-tasks

**Context:**  
JIRA Epic Link and parent fields require the parent to exist before the child.

**Decision:** Sort artifacts by type before committing: `Epic → Story → Task → Bug → Spike → Sub-task`

**Implementation:**

```typescript
const ORDER = ["Epic", "Story", "Task", "Bug", "Spike", "Sub-task"];
const sorted = [...candidates].sort(
  (a, b) => ORDER.indexOf(a.type) - ORDER.indexOf(b.type),
);
```

A `refToKey` map tracks `{draftRef → jiraKey}` as each issue is created.

**Consequences:**

- Epics must be in the same draft as their children for auto-linking to work
- Alternatively, `epicLinkKey` or `parentKey` can point to existing JIRA issues

## Decision 8: Delete Requires "DELETE CONFIRMED" Phrase

**Context:**  
Accidental deletion of JIRA issues is difficult to reverse (no recycle bin in Data Center by default).

**Decision:** `jira_delete_issue` requires `confirmation_phrase: "DELETE CONFIRMED"` exactly.

**Rationale:**

- Makes deletion a conscious, explicit action
- Prevents Claude from deleting issues when processing ambiguous user intent
- Simple pattern — no external confirmation system needed

**Consequences:**

- Slightly more verbose API call for intentional deletions
- Cannot be bypassed by the AI without user explicitly including the phrase
